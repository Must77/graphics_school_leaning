《数字图像处理》结课实验报告
题目：BMP 图像水印实验（C++11/14，底层 API）

摘要
本实验基于 C++11/14 与底层文件接口实现了 BMP 图像的文字水印添加功能。程序内置 8x8 像素位图字体，支持 ASCII 可见字符的渲染，通过 Alpha 混合算法将水印文字以可控不透明度叠加到原始图像上。水印放置支持五种定点位置（居中、四角）和全图平铺两种模式，字体大小根据图像尺寸自动缩放以保持可读性。实验在三种不同分辨率的测试样例上验证了四种水印配置，结果表明程序功能正确、运行稳定，水印视觉效果清晰且不遮挡原图主体内容。

引言
数字水印是图像版权保护和信息溯源的常用手段，广泛应用于图片分发、文档安全和内容认证等场景。可见水印通过在图像表面叠加半透明文字或图案，使观者能够识别图像来源或版权归属，同时不完全遮挡原始内容。从实现角度看，可见文字水印涉及字体光栅化、Alpha 混合、位置计算和缩放控制等技术点。本实验以底层 BMP 读写为基础，实现不依赖任何外部字体库和图像处理库的纯 C++ 文字水印方案，旨在通过工程实践加深对像素级图像操作和混合算法的理解。

一、实验内容
1. 实验目标与范围
本实验的目标是实现图像水印子函数，能够在 BMP 图像上叠加指定文本的半透明水印。功能需求包括：支持自定义水印文本内容、不透明度和放置位置；提供居中、四角定点和全图平铺两种放置策略；字体大小根据图像分辨率自动适配。输入支持 24 位和 32 位无压缩 BMP，输出统一为 24 位 BMP。程序通过命令行参数控制所有配置，便于批量测试和自动化验证。

2. 方法对比与选择依据
文字水印的实现方式有多种路线。第一种是调用系统字体渲染接口（如 Windows GDI 或 FreeType），优点是支持 TrueType 字体和丰富的排版控制，缺点是引入外部依赖且跨平台移植复杂。第二种是使用预渲染的字体纹理图片作为水印源，灵活性较好但需要额外的资源文件管理。第三种是内置像素位图字体，将每个字符的像素模式直接编码在源代码中，无任何外部依赖，适合教学演示和轻量级场景。本实验选择第三种方案，采用 8x8 像素的内置位图字体，覆盖 ASCII 可打印字符，通过整数倍缩放适配不同分辨率，实现零依赖、可移植的水印功能。

二、实现方法
1. 模块与接口设计
程序在 BMP 读写框架基础上新增三个功能模块：字体数据模块、文本渲染模块和水印放置模块。字体数据模块以静态数组形式存储 91 个 ASCII 字符（0x20 至 0x7A）的 8x8 像素位图，每个字符用 8 个字节表示，每字节的 8 个比特位对应一行像素的亮灭状态。文本渲染模块负责将字符串逐字符查表并绘制到图像指定坐标，支持整数倍缩放和 Alpha 混合。水印放置模块根据位置参数计算起始坐标：center 为图像中心，tl/tr/bl/br 分别为四角（带边距），tile 模式则以固定间距在全图范围内重复渲染，相邻行做半周期偏移以避免视觉对齐感过强。程序命令行接口为"bmp_watermark 输入路径 水印文本 输出路径 [不透明度] [位置]"，不透明度默认 0.3，位置默认 center。

2. Alpha 混合算法
水印叠加的核心是 Alpha 混合。对于水印像素覆盖的每个位置，输出颜色由公式 C_out = C_bg * (1 - α) + C_wm * α 计算，其中 C_bg 为原始像素颜色，C_wm 为水印颜色（本实验固定为白色），α 为用户指定的不透明度。当 α = 0 时水印完全透明，原图不受影响；当 α = 1 时水印完全不透明，覆盖原始内容。实际使用中通常取 0.2-0.5 的范围，使水印可见但不严重遮挡图像内容。混合运算对 R、G、B 三通道分别独立进行，结果四舍五入并裁剪到 [0, 255] 范围。

3. 内置位图字体与自动缩放
程序内置的 8x8 位图字体覆盖了空格、数字、大小写字母和常用标点符号。每个字符存储为 8 个 uint8_t 值，每个值的最高位对应最左像素。渲染时按从左到右顺序排列字符，每个字符占 8 * scale 个像素宽度。缩放因子 scale 由图像较短边除以 128 自动计算，确保水印在小图上不会过大、在大图上不会过小。例如 256x256 图像的 scale 为 2（字符 16x16 像素），1920x1080 图像的 scale 为 8（字符 64x64 像素），均能保持良好的可读性。

4. 平铺水印的布局策略
平铺模式（tile）用于在整幅图像上重复叠加水印文字，常见于商业图库的版权保护。布局策略为：以固定垂直间距逐行排列，每行以固定水平间距重复渲染水印文本；偶数行与奇数行做半个水平周期的偏移，形成交错排列效果，避免所有水印上下对齐带来的视觉呆板感。间距参数根据文本渲染宽度和字符高度自动计算，保证水印密度适中。

5. 可复现实验步骤
编译命令在 Linux 下为 g++ -std=c++14 -O2 -Wall -Wextra -o bmp_watermark bmp_watermark.cpp，在 Windows MSVC 下为 cl /std:c++14 /EHsc /W4 /nologo bmp_watermark.cpp /Fe:bmp_watermark.exe。运行示例：bmp_watermark input.bmp "SAMPLE" output.bmp 0.3 center 在图像中心叠加不透明度 0.3 的水印；bmp_watermark input.bmp "WATERMARK" output.bmp 0.3 tile 生成全图平铺水印。自动化测试通过 auto_bmp_benchmark.py 生成三组测试 BMP 并运行四种水印配置，结果输出至 benchmark_data.md；报告图片通过 generate_report_images.py 生成对比图和耗时柱状图。

三、实验结果与分析
1. 实验环境
实验在以下环境中完成：CPU 为 Intel Core i5-3230M（基础频率 2.60GHz，最大睿频 3.20GHz），内存 8GB DDR3，显卡为 AMD Radeon HD 7650M（独显）与 Intel HD Graphics 4000（核显），操作系统为 Ubuntu 25.10（Linux 内核 6.17.0），编译器为 GCC 15.2.0，编译标准 C++14，优化等级 -O2。代码无外部依赖，可在任何支持 C++14 的编译器上直接编译运行。

2. 结果对比说明
每组测试样例生成了"原图 — 居中水印(α=0.3) — 平铺水印(α=0.3) — 右下角水印(α=0.5) — 居中水印(α=0.7)"五图并排的对比图。居中水印在图像中央显示清晰的白色文字，α=0.3 时原图内容透过水印仍然可辨，α=0.7 时水印更为醒目但原图信息有所遮挡。平铺模式下水印文字均匀分布于整幅图像，交错排列视觉效果自然。右下角定点水印仅影响图像局部区域，适合版权标识的非侵入式放置。在三组不同分辨率样例中，字体自动缩放均能保持水印文字清晰可读，说明缩放策略有效。

3. 量化结果与性能分析
从自动化数据看，所有 12 组测试（3 种分辨率 × 4 种配置）均返回码为 0，处理正常。小尺寸样例（256x256）各配置耗时约 0-1 ms；中尺寸（1024x768）约 8-9 ms；大尺寸（1920x1080）约 20-25 ms。同一分辨率下不同配置的耗时差异很小，说明水印渲染的计算开销远低于 BMP 读写的 I/O 开销，整体性能瓶颈在文件操作而非像素混合。灰度统计方面，添加白色水印后灰度均值较原图略有升高（因白色像素混入），不透明度越高升幅越明显，符合 Alpha 混合的数学预期。

4. 局限性与改进思考
当前实现的主要局限有三点：第一，内置位图字体仅覆盖 ASCII 字符，不支持中文等多字节字符，如需中文水印需引入字体文件解析；第二，水印颜色固定为白色，在浅色图像上可能不够醒目，后续可增加颜色参数或自适应颜色选择；第三，当前仅实现可见水印，未涉及不可见数字水印（如 DCT 域嵌入），后者在版权保护中具有更强的鲁棒性。若进一步扩展，可加入旋转文字水印（结合 sub1 的旋转功能）实现斜向排列效果。

四、结论与体会
1. 结论
本实验在底层 BMP 读写框架上成功实现了文字水印子函数，支持定点和平铺两种放置模式、可控不透明度和自动缩放。程序在三组分辨率样例的四种配置下通过了功能验证和性能测试，水印视觉效果清晰、Alpha 混合正确、运行性能稳定，满足实验要求。

2. 心得体会
本次实验让我对像素级图像操作有了更细致的理解，特别是 Alpha 混合虽然公式简单，但在工程实现中需要注意数值裁剪和四舍五入以避免溢出。内置位图字体的设计过程也加深了我对字符光栅化原理的认识：每个可见字符本质上就是一张微小的二值图像，渲染过程就是将这些微图按顺序绘制到目标画布上。此外，通过对比不同不透明度的水印效果，我对视觉感知与数值混合之间的关系有了更直观的体会，这对后续图像增强和融合类实验具有参考价值。
